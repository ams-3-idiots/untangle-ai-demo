# 커밋·PR 워크플로

## 적용 범위

git에 쓰기를 발생시키는 모든 작업 — 브랜치 생성, 커밋, push, PR 생성·병합.

## 핵심 규칙 요약

| ID | 규칙 |
| --- | --- |
| W-1 | 브랜치 생성·커밋·push·PR 생성/병합 등 **모든 git 쓰기 작업은 실행 전에 사용자에게 내용을 제시하고 승인을 받는다.** 승인 없이 실행하지 않는다. |
| W-2 | `main`에 직접 커밋·push 하지 않는다. 항상 기능 브랜치에서 작업하고 PR로 병합한다. |
| W-3 | 커밋·PR 메시지는 **전부 한국어**로 쓴다. (타입 접두어 없이 한국어 문장) |
| W-4 | 한 커밋은 하나의 논리적 변경만 담는다. 무관한 변경을 섞지 않는다. |
| W-5 | 비밀키·자격증명·`.env`·`secrets.toml`은 **절대 커밋하지 않는다.** |

## 1. 시작 전

작업할 브랜치에 이미 이동해있다고 가정하고 작업을 진행.

## 2. 브랜치 전략

- `main` 직접 작업 금지. 모든 변경은 기능 브랜치에서 한다.
- 브랜치 이름: `<구분>/<F-번호>-<짧은-설명>` 형식.
  (메시지만 한국어이고, 브랜치 이름은 영문 슬러그를 쓴다.)
  - 구분: `feat`(기능), `fix`(버그), `docs`(문서), `refactor`, `chore`
  - 예: `feat/F2-brain-dump`, `fix/F6-complete-toggle`, `docs/workflow`
- 브랜치는 최신 `main`에서 분기한다.
  ```
  git switch main
  git pull
  git switch -c feat/F2-brain-dump
  ```

## 3. 커밋

형식(접두어 + 한국어 메시지 — `접두어: <메시지>`):

```
<접두어>: <무엇을 했는지 한 줄 요약 — 50자 내외, 마침표 없음>

<선택: 왜 / 무엇을 어떻게 바꿨는지>
관련: F2-2, F2-3 (충족한 AC 명시)

Co-Authored-By: <에이전트 이름> <메일>
```

접두어는 2. 브랜치 전략과 동일하게 맞춘다.

| 접두어 | 사용 |
| --- | --- |
| `feat` | 기능 추가·변경 (F-번호 기능 구현) |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 (`docs/`, `README`, `AGENTS.md`) |
| `refactor` | 동작 변화 없는 코드 구조 개선 |
| `chore` | 의존성·설정·빌드 등 (`pyproject.toml`, `uv.lock`, `.gitignore`) |

- 접두어는 소문자로 쓰고, 콜론 뒤에 한 칸을 둔다. (`feat: 메시지`)
- 메시지 본문은 전부 한국어로 쓴다. (W-3)
- 제목은 요약형으로 한 줄. 무엇이 바뀌는지 한눈에 드러나게 쓴다.
- 본문에는 변경 내용을 한국어로 서술한다.
- 의존성을 바꿨다면 `uv.lock`도 같은 커밋에 포함한다.

예시:

```
브레인덤프 할 일 추출 기능 추가

자유 입력에서 실행 가능한 할 일 후보를 5개 이하로 추출한다.
추출이 어려운 추상적 입력은 빈 결과와 재입력 안내를 반환한다.
```

## 4. 비밀키·민감 정보

- OpenAI/Claude **API 키 등 자격증명은 코드·커밋에 절대 포함하지 않는다.** (W-5)
- 키는 환경 변수 또는 `st.secrets`(`.streamlit/secrets.toml`)로 주입한다.
- 커밋 전 `.env`, `.streamlit/secrets.toml`이 `.gitignore`에 포함됐는지 확인한다. 빠져 있으면 임의로 진행하지 말고 사용자에게 알린다.
- 커밋 직전 `git diff --staged`로 키가 섞이지 않았는지 점검한다.

## 5. PR 절차

- `gh` CLI를 사용하고 base는 `main`으로 한다.
- PR 생성 전, W-1에 따라 **제목·본문을 사용자에게 보여주고 승인을 받은 뒤** 실행한다.
- 제목: 한국어 한 줄 요약.
- 본문 템플릿:

```
## 요약
<무엇을 왜 바꿨는지>

## 관련 요구사항
- 구현·변경한 F-번호: F?-?, F?-? (docs/FEATURES.md 기준)

## 변경 사항
- ...

## 확인 방법
- `uv run streamlit run app.py` 실행 후 <대상 화면/동작> 확인

## 요구사항 충족 체크리스트
- [ ] 이 PR이 다루는 F-번호를 모두 적었다. (위 '관련 요구사항')
- [ ] 각 F-번호의 인수 기준(AC: F?-1, F?-2 …)을 docs/FEATURES.md와 대조해 모두 충족했다.
- [ ] 미충족·보류한 AC가 있다면 그 사유를 본문에 적었다.
- [ ] GP-1 준수: AI 출력이 사용자 확정 전 실제 데이터에 반영되지 않는다. (AI 기능 변경 시)
- [ ] AC 외 동작 변경이 기존 다른 F-번호 요구사항을 깨지 않는지 확인했다.

## 체크리스트
- [ ] 비밀키·.env 미포함
- [ ] 의존성 변경 시 uv.lock 동기화
```

- **요구사항 충족 점검 방법**: PR이 다루는 각 F-번호를 `docs/FEATURES.md`에서 찾아, 해당 기능의 모든 AC(`F?-1`, `F?-2` …)를 한 줄씩 대조한다. 하나라도 충족되지 않으면 PR 본문에 그 사유를 명시하거나, 충족 후 다시 올린다.

- 명령 예:
  ```
  gh pr create --base main --title "<한국어 제목>" --body "<위 템플릿>"
  ```

## 6. 모든 단계 사용자 확인 (W-1 상세)

에이전트는 다음 각 단계를 **실행하기 전에** 내용을 제시하고 사용자 승인을 받는다.

| 단계 | 사용자에게 보여줄 것 |
| --- | --- |
| 브랜치 생성 | 브랜치 이름·분기 기준 |
| 커밋 | 스테이징 목록·커밋 메시지 전문 |
| push | 대상 브랜치·원격 |
| PR 생성/병합 | PR 제목·본문 전문, base 브랜치 |

승인 없이 실행하지 않으며, 승인 범위를 넘는 추가 작업도 다시 확인한다.

## 7. 금지·주의

- `main` 직접 커밋·push 금지. (W-2)
- force-push, rebase로 공유 히스토리 재작성, `git reset --hard`, `--no-verify`는 사용자가 명시적으로 요청할 때만. (W-7)
- 검증: 현재 테스트 프레임워크가 없다. 최소한 `uv run streamlit run app.py`로 앱이 뜨고 변경한 화면이 동작하는지 스모크 확인한다.

## 8. 작업 후 자가 점검

커밋·PR 전에 스스로 확인한다.

- [ ] 비밀키·민감 정보가 포함되지 않았는가, `uv.lock`이 동기화됐는가
- [ ] 커밋·PR 메시지가 한국어이고 관련 요구사항을 명시하는가
- [ ] 모든 git 쓰기 작업에 대해 사용자 승인을 받았는가 (W-1)
